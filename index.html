Level 1: Fundamentals — Understanding CrowdStrike
What is CrowdStrike Falcon?
CrowdStrike Falcon is a cloud-native endpoint protection platform (EPP) that combines next-gen antivirus (NGAV), endpoint detection & response (EDR), threat intelligence, and managed hunting in a single lightweight agent (the Falcon Sensor).

Key Components:
• Falcon Sensor — Lightweight agent installed on endpoints (Windows, Linux, Mac). Uses ~1% CPU, ~40MB RAM.
• Falcon Console — Web-based management UI at falcon.crowdstrike.com
• Threat Graph — Cloud AI that processes 1 trillion+ events/day for real-time detection
• CID — Customer ID, unique to your organization

Core Modules:
• Falcon Prevent — NGAV (next-gen antivirus) — replaces traditional AV
• Falcon Insight — EDR (endpoint detection & response) — visibility + investigation
• Falcon OverWatch — 24/7 managed threat hunting by CrowdStrike experts
• Falcon Discover — IT hygiene and asset inventory
• Falcon Intelligence — Threat intel reports and IOC feeds
• Falcon LogScale — Log management and SIEM (formerly Humio)
• Falcon Identity — Identity threat protection (Active Directory monitoring)
Level 2: Detection — IOA vs IOC (The Most Important Concept)
IOC (Indicator of Compromise) — Static indicators: file hashes, IP addresses, domain names.
• Reactive: you need to know the indicator first
• Easy to evade: attacker changes hash → bypass
• Example: Block SHA256 of known Mimikatz binary
• Use case: Block known-bad from threat intel feeds

IOA (Indicator of Attack) — Behavioral patterns: what processes do, not what they look like.
• Proactive: detects new/unknown attacks by behavior
• Hard to evade: behavior must happen for attack to work
• Example: "Any process accessing LSASS memory that isn't a known OS process"
• Use case: Detect zero-day credential theft tools

✅ Key Insight: IOAs are more powerful because they detect the technique, not the tool. A Mimikatz hash can change, but LSASS memory access behavior cannot.

CrowdStrike Detection Priority:
1. Machine Learning (ML) — Automated malware detection
2. IOA Rules — Behavioral detection (MOST IMPORTANT to learn)
3. IOC Policies — Known-bad blocking
4. Custom IOA Rules — Your custom detections (this is what we'll build!)
IOA = Behavioral IOC = Static IOA > IOC for detection
Level 3: Policy Types — What You Can Configure
1. Prevention Policies (Console > Configuration > Prevention Policies)
Controls what the sensor blocks automatically. Sliders for:
• NextGen AV sensitivity (Cautious → Moderate → Aggressive → Extra Aggressive)
• Cloud ML, Sensor ML, Adware/PUP, Script-Based Execution Monitoring
• Behavior-based protection: Credential dumping, lateral movement, ransomware
• Exploit mitigation: Stack pivot, heap spray, SEH overwrite

2. Custom IOA Rule Groups (Console > Custom IOA Rule Groups)
Your custom behavioral detections. Types:
• Process Creation — trigger on process name, command line, parent process
• Process Injection — trigger on memory injection techniques
• DNS Request — trigger on specific domain queries
• Network Connection — trigger on connections to specific IPs/ports
• File Creation/Modification — trigger on suspicious file operations
• Registry Operations — trigger on persistence registry keys

3. IOC Management (Console > IOC Management)
Import/manage static indicators:
• SHA256 hashes — block/detect specific files
• MD5 hashes — legacy hash blocking
• Domains — block C2/malicious domains
• IPv4/IPv6 — block malicious IPs
• Actions: Prevent, Detect, or Allow (whitelist)

4. Firewall Rules (Console > Network Protection > Firewall Management)
Host-based firewall rules pushed via sensor:
• Block/Allow/Monitor by IP, port, protocol, direction
• Rule Groups assigned to Firewall Policies

5. Real Time Response (RTR) (Console > Investigate > select host > RTR)
Remote shell access to endpoints for live investigation:
• Read-only commands: ps, netstat, filelist, reg query
• Active Responder: kill, delete, remediate
• Custom scripts: PowerShell/Bash scripts pushed to endpoints

Level 4: Severity & Disposition — How CrowdStrike Handles Detections
Severity Levels:
• Critical — Active breach, immediate response needed (ransomware, credential theft)
• High — Likely malicious, needs investigation (suspicious execution, C2)
• Medium — Potentially unwanted, context needed (adware, PUP)
• Low/Informational — Visibility event, no action required

Disposition (What Happens When Rule Triggers):
• Block, Block (ID: 80) — Kill process + prevent execution. Use for critical threats.
• Detect, Detect (ID: 60) — Alert only, don't block. Use when testing new rules.
• Monitor (ID: 40) — Log silently. Use for visibility/baselining.

✅ Best Practice: Always start new Custom IOA rules in Detect mode first. Monitor for 1-2 weeks for false positives. Then promote to Block after tuning.

🛡 Custom IOA Rule Builder — Hands-On Lab

What is a Custom IOA Rule?
Custom IOA (Indicator of Attack) rules let you create behavioral detections that trigger when specific process behaviors are observed on endpoints. They match on process creation, command lines, parent-child relationships, network connections, DNS requests, and more.

How to create in Falcon Console:
1. Go to Configuration > Custom IOA Rule Groups
2. Click "Create Rule Group" — name it (e.g., "Credential Theft Detection")
3. Select Platform: Windows / Linux / Mac
4. Click "Add New Rule"
5. Choose Rule Type: Process Creation (most common)
6. Configure field values (Image, CommandLine, ParentImage, etc.)
7. Set Disposition: Detect (start here!) or Block
8. Set Severity: Critical / High / Medium / Low
9. Enable the rule
10. Assign Rule Group to Prevention Policy — this is the step people forget!

🔎 IOC Policy Builder — Hands-On Lab

What is an IOC Policy?
IOC (Indicator of Compromise) policies let you import known-bad indicators (hashes, IPs, domains) and tell CrowdStrike what to do when they're seen on endpoints.

Indicator Types: SHA256, MD5, Domain, IPv4, IPv6
Actions: Prevent (block), Detect (alert), Allow (whitelist)
How to add in Console: IOC Management > Add Indicator
API Endpoint: POST /iocs/entities/indicators/v1

✅ Pro Tip: Set expiration dates on IOCs (90 days default). Threat infrastructure changes — stale IOCs cause false positives.

🛡 Prevention Policy Builder — Hands-On Lab

What is a Prevention Policy?
Prevention Policies control the sensor's built-in protection capabilities. They're assigned to Host Groups and determine how aggressively the sensor blocks threats.

Key Settings:
• NextGen AV: Cloud ML + Sensor ML detection sensitivity sliders
• Ransomware: Shadow copy protection, volume encryption blocking
• Script Monitoring: PowerShell, WScript, VBScript, AMSI integration
• Exploit Mitigation: DEP, ASLR enforcement, heap spray prevention
• Credential Protection: LSASS read protection, credential dumping blocking

Console Path: Configuration > Prevention Policies > Clone or Create
✅ Best Practice: Never modify the "platform_default" policy. Clone it and create role-specific policies (Servers, Workstations, High-Value Assets).

CrowdStrike Host Firewall Rules
Falcon Firewall Management pushes host-based firewall rules to endpoints via the sensor. Rules are organized in Rule Groups, which are assigned to Firewall Policies.

Console Path: Network Protection > Firewall Management > Rule Groups
Rule Parameters: Direction (IN/OUT), Protocol (TCP/UDP/ICMP), Port, Remote Address, Action (ALLOW/DENY/MONITOR)
✅ Tip: Put DENY rules before ALLOW rules. Rules are evaluated top-to-bottom.

💻 Real Time Response (RTR) — Hands-On Lab

What is RTR?
Real Time Response gives you a remote shell on any managed endpoint for live investigation and remediation — without needing VPN, RDP, or physical access.

Console Path: Investigate > Host Search > Select Host > Connect (RTR icon)

Permission Levels:
• Read-Only Analyst: ps, netstat, filelist, reg query, users, ifconfig
• Active Responder: kill, rm, reg delete, remediate, memdump
• RTR Admin: put files, run scripts, get files, custom scripts

✅ Key Commands: ps (processes), netstat (connections), filelist (directory listing), reg query (registry), runscript (execute script)

📊 Falcon LogScale Query Builder — Hands-On Lab

LogScale Query Language Basics
Falcon LogScale (formerly Humio) uses its own query language for searching and analyzing endpoint telemetry.

Key Concepts:
• #event_simpleName — Event type filter (ProcessRollup2, DnsRequest, NetworkConnectIP4, etc.)
• | — Pipe operator (chain transformations)
• field=value or field=/regex/ — Field matching
• groupBy() — Aggregate results by field
• count(), avg(), sum() — Aggregation functions
• sort() — Order results
• test() — Post-aggregation filtering

Common Event Types:
• ProcessRollup2 — Process creation events (most used!)
• DnsRequest — DNS queries from endpoints
• NetworkConnectIP4 — Outbound network connections
• UserLogon — User login events
• FileWritten — File creation/modification
• RegKeyValueSet — Registry modifications
• ModuleLoad — DLL loading events

⚙ Falcon API — Automation Lab

CrowdStrike Falcon API Basics
The Falcon API lets you automate everything you can do in the console. Authentication uses OAuth2 client credentials flow.

Setup:
1. Support & Resources > API Clients and Keys
2. Create new API Client with required scopes
3. Note your client_id and client_secret
4. Base URL: https://api.crowdstrike.com (US-1), api.us-2.crowdstrike.com (US-2), api.eu-1.crowdstrike.com (EU)

Authentication: POST to /oauth2/token with client_id + client_secret → get bearer token

Key Endpoints:
• GET /devices/queries/devices/v1 — List hosts
• POST /detects/queries/detects/v1 — Query detections
• POST /iocs/entities/indicators/v1 — Create IOCs
• POST /ioarules/entities/rules/v1 — Create IOA rules
• POST /devices/entities/devices-actions/v2 — Contain/lift containment
• POST /real-time-response/entities/sessions/v1 — Start RTR session
