📖 POC Day-by-Day Setup Guide
Complete step-by-step guide - follow this to set up CrowdStrike from zero to hero
🎯 Before You Start - What You Need
FROM YOUR CROWDSTRIKE PARTNER/SE: 1. Falcon Console URL (e.g., falcon.crowdstrike.com) 2. Admin credentials (email + password) 3. Customer ID (CID) - found in Console > Host Setup 4. API Client ID & Secret (if using API) 5. Sensor installer files (or download from console) FROM THE CLIENT: 1. Number of endpoints to deploy on (start with 10-50) 2. OS types (Windows, macOS, Linux) 3. Network requirements (proxy, firewall whitelist) 4. Existing security tools (for coexistence) 5. Key applications to test against (ensure no FPs) 6. Compliance requirements (HIPAA, PCI, SOC2, etc.) WHITELIST REQUIREMENTS: Client firewall must allow outbound HTTPS (443) to: • ts01-b.cloudsink.net • lfodown01-b.cloudsink.net • *.crowdstrike.com
📅 DAY 1: Console Setup & First Deployment (2-3 hours)
STEP 1: Login to Falcon Console → Go to falcon.crowdstrike.com (or your region's URL) → Login with admin credentials → You'll land on the main dashboard STEP 2: Download Sensor Installer → Console > Support & Resources > Sensor Downloads → Download installer for each OS type → Note the Customer ID (CID) shown on the download page STEP 3: Create Host Groups (IMPORTANT - DO THIS FIRST) → Console > Host Setup > Host Groups → Click "Create Host Group" → Create these groups: • "POC-Test-Group" (Dynamic: hostname contains 'POC') • "POC-Detect-Only" (Dynamic: tag contains 'detect-mode') • "POC-Full-Prevention" (Dynamic: tag contains 'full-prevent') → These let you apply different policies to different hosts STEP 4: Create Prevention Policy (DETECT ONLY for Day 1) → Console > Configuration > Prevention Policies → Click "Create Policy" → Name: "POC-Detect-Mode" Platform: Windows → Set ALL toggles to DETECT (not prevent yet!) → Cloud ML: Moderate, Sensor ML: Moderate → Assign to "POC-Detect-Only" host group → This ensures no business disruption on Day 1 STEP 5: Install Sensor on First 5 Endpoints → Copy installer + CID to target machines → Windows: WindowsSensor.exe /install /quiet CID=YOUR_CID GROUPING_TAGS="poc,detect-mode" → Wait 5 minutes for hosts to appear in console → Verify: Console > Hosts > search for your endpoints → Check sensor version and "Connected" status STEP 6: Verify Deployment → Console > Host Setup > Sensor Health → All hosts should show green "Connected" status → If any show "RFM" = Reduced Functionality Mode (troubleshoot!) → Run EICAR test: create eicar.com file on one host → Check Console > Activity > Detections for EICAR alert
📅 DAY 2-3: Expand Deployment & Baseline (2-4 hours)
STEP 7: Deploy to Remaining POC Endpoints → Deploy sensor to all POC endpoints (target: 20-50) → Use same GROUPING_TAGS="poc,detect-mode" for all → For bulk deployment: • GPO: Copy to NETLOGON share, create startup script • SCCM: Create Application with detection rule • Ansible: Use playbook from Deployment section → Monitor Console > Hosts for new endpoints appearing STEP 8: Review Baseline Detections → After 24-48 hours in DETECT mode: → Console > Activity > Detections → Review all detections - categorize as: • True Positive: Real threat activity (good!) • False Positive: Legitimate app flagged (needs exclusion) • Noise: Low-severity info detections (normal) → Note any false positives for exclusion configuration STEP 9: Configure Exclusions (if needed) → Console > Configuration > Prevention Policies > Exclusions → For each false positive, add: • File path exclusion (e.g., C:\CompanyApp\app.exe) • Hash exclusion (allow specific file hash) → Keep exclusions minimal - better to tune ML than exclude STEP 10: Set Up Detection Policy → Console > Configuration > Detection Policies → Create "POC-Detection-Policy" → Cloud ML: Moderate, Sensor ML: Moderate → Extended Visibility: ON (more telemetry data) → Assign to your POC host groups
📅 DAY 4-5: Enable Prevention & Custom Rules (3-4 hours)
STEP 11: Create Full Prevention Policy → Console > Configuration > Prevention Policies → Create "POC-Full-Prevention" → Enable ALL behavioral protections in PREVENT mode: ✓ Credential Dumping, Ransomware, Code Injection ✓ Lateral Movement, Webshell, Fileless Ransomware → ML: Moderate (or Aggressive if client wants max protection) → Exploit Mitigation: ALL ON → Script Monitoring: PowerShell + cmd + WMI = PREVENT → Assign to "POC-Full-Prevention" host group → Move a few test hosts to this group to validate STEP 12: Create Custom IOA Rules → Console > Configuration > Custom IOA Rule Groups → Create group: "POC-Custom-Rules" (Windows) → Add rules from this toolkit's IOA section: 1. Mimikatz Detection (T1003.001) 2. Encoded PowerShell (T1059.001) 3. Ransomware Shadow Delete (T1490) 4. PsExec Lateral Movement (T1021.002) 5. LOLBin Abuse (T1218) → Start with DETECT disposition → Assign group to your POC host groups STEP 13: Import IOC Blocklist → Console > Support > IOC Management → Click "Upload Indicators" → Import known-bad hashes, domains, IPs from: • AlienVault OTX (free) • abuse.ch (MalwareBazaar, URLhaus) • Your own threat intel → Use this toolkit's IOC section to generate import JSON STEP 14: Configure Firewall Rules → Console > Configuration > Firewall Management → Create rule group: "POC-Firewall-Rules" → Add rules: • Block C2 ports (4444, 5555, 8080, 8443) • Block external SMB (445, 139) • Block Tor (9001, 9030, 9050) → Assign to POC host groups STEP 15: Configure USB Device Control → Console > Configuration > USB Device Control → Create policy: "Block-USB-Storage" → Block all mass storage devices → Add exceptions for approved devices if needed → Assign to POC host groups
📅 WEEK 2-3: Testing & Validation (4-6 hours)
STEP 16: Run Test Scenarios → Go to "POC Test Scenarios" section in this toolkit → Execute each of the 15 test scenarios → Mark Pass/Fail for each → Key tests to impress client: 1. EICAR test (basic - should pass immediately) 2. Mimikatz (shows credential protection) 3. Ransomware simulation (biggest client fear) 4. PowerShell encoded command (fileless attack) 5. USB block (visual proof of device control) 6. Network containment (impressive response capability) STEP 17: Demonstrate RTR (Real Time Response) → Console > Hosts > Select any host > Connect (RTR) → Show the client you can: • See running processes: ps • Check network connections: netstat • Kill suspicious processes: kill • Collect evidence: runscript • Download files: get → This is a HUGE differentiator vs competitors STEP 18: Show Threat Hunting → Console > Investigate > Event Search → Run LogScale queries from this toolkit → Show proactive hunting capabilities → Demonstrate: "We can find threats others miss"
📅 WEEK 4: Report & Close (2-3 hours)
STEP 19: Generate POC Report → Go to "Report Generator" section in this toolkit → Fill in client info, test results, metrics → Generate executive summary + detailed report → Include ROI calculation (use the built-in calculator) STEP 20: Client Presentation → Show detection rate (target: >95%) → Show prevention rate (target: >90%) → Show mean time to detect (<5 seconds) → Walk through 2-3 best detection examples → Show ROI: CrowdStrike saves $X per year → Recommend production deployment plan STEP 21: Transition to Production → If client approves (PO!): → Move all hosts to full prevention policy → Increase ML to Aggressive → Enable all IOA rules in Prevent mode → Set up SIEM integration → Configure email notifications → Enable Falcon OverWatch (managed hunting)

🔍 How to Investigate a Detection (Step by Step)
WHEN YOU SEE A DETECTION ALERT: 1. TRIAGE (30 seconds) → Activity > Detections → Look at: Severity + Technique + Hostname → Critical + Credential Access on server = DROP EVERYTHING → Low + Info on workstation = can wait 2. INVESTIGATE THE PROCESS TREE (2-3 minutes) → Click on the detection → View "Execution Details" / Process Tree → Look at: • What process ran? (powershell, cmd, unknown.exe?) • What launched it? (explorer = user ran it, w3wp = webshell) • What did it do? (network connection? file write? registry?) • What was the command line? (encoded? suspicious args?) 3. CHECK THE HOST (1-2 minutes) → Click on hostname in detection → Check: Is it a critical server or user workstation? → Check: Is the user who ran it legitimate? → Check: Are there other recent detections on this host? 4. DECIDE: CONTAIN OR MONITOR → If active threat (C2 connection, lateral movement): → CONTAIN THE HOST immediately → Host > Actions > Contain → This blocks all network except CrowdStrike cloud → If suspicious but not confirmed: → Set detection to "In Progress" → Monitor for additional activity 5. COLLECT EVIDENCE (via RTR) → Host > Connect to RTR → Run: ps (check running processes) → Run: netstat (check connections) → Run: runscript -CloudFile="CollectForensics" → Run: get C:\evidence.zip (download artifacts) 6. REMEDIATE → Kill malicious processes: kill PID → Remove persistence: check registry run keys, scheduled tasks → Quarantine files: rm suspicious_file.exe → Reset compromised credentials (via AD admin) 7. CLOSE THE LOOP → Update detection status to "Resolved" or "Closed" → Document findings → If new IOC found: add to IOC blocklist → If new attack pattern: create Custom IOA rule → Lift containment when remediation confirmed

❓ Client FAQ & Objection Handling
Common client questions and how to answer them confidently

"How is CrowdStrike different from our current antivirus?"

ANSWER: Traditional AV uses signatures - a database of known bad files. It can only detect what it already knows. CrowdStrike uses Machine Learning and Behavioral Analysis: • ML: Analyzes file characteristics to determine if it's malicious, even for never-seen-before threats • Behavioral: Watches WHAT programs DO, not just what they look like → If a process deletes shadow copies + encrypts files = ransomware behavior = blocked → Doesn't matter if it's a brand new ransomware variant Plus: EDR (investigation), RTR (remote response), Threat Intel, Managed Hunting Traditional AV = just a lock on the door CrowdStrike = lock + cameras + alarm + 24/7 security guard

"Will it slow down our computers?"

ANSWER: CrowdStrike is specifically designed to be lightweight: • CPU: ~1% average usage (less than Chrome) • RAM: ~50-80MB (less than Outlook) • Disk: ~25MB installed • No scheduled full-disk scans (unlike traditional AV) • No bulky signature database downloads During our POC, we monitored [N] endpoints for [X] weeks with zero performance complaints. Your users won't notice it's there. Fun fact: Many customers report BETTER performance after removing legacy AV and replacing with CrowdStrike because CrowdStrike's sensor is more efficient than traditional AV scanning engines.

"What about false positives? Will it block our business apps?"

ANSWER: False positive management is a key CrowdStrike strength: 1. ML Tuning: We start at "Moderate" sensitivity during POC. If any FPs occur, we can adjust per-app or globally. 2. Detect-First Approach: We start in detect-only mode. We validate zero FPs BEFORE enabling prevention. Only after confirming safety do we enable blocking. 3. Granular Exclusions: We can exclude specific: • File paths (e.g., your custom app folder) • File hashes (specific versions of your apps) • Process names (named exclusions) During our POC: [0] false positives on your production applications. Your critical apps ran without any interference.

"What happens if the internet goes down?"

Management requires connectivity.

🔧 Troubleshooting Guide
Common issues during POC deployment and how to fix them

CRITICAL Sensor shows RFM (Reduced Functionality Mode)

CAUSE: Kernel driver failed to load. Common on: • Older kernels (Linux) • Secure Boot conflicts • VBS/HVCI enabled Windows FIX: 1. Check kernel compatibility: falconctl -g --rfm-reason 2. Update sensor to latest version 3. Linux: Check kernel version against support matrix 4. Windows: Temporarily disable VBS for testing 5. If persists: Contact CrowdStrike support with rfm-reason output VERIFY: Windows: reg query "HKLM\SYSTEM\CrowdStrike" /v RFMState Linux: /opt/CrowdStrike/falconctl -g --rfm-state Expected: rfm-state=false

HIGH Sensor not connecting to cloud

CAUSE: Network connectivity blocked FIX: 1. Verify outbound HTTPS (443) is allowed to: • ts01-b.cloudsink.net • lfodown01-b.cloudsink.net • *.crowdstrike.com 2. Check proxy settings: falconctl -g --aph (proxy host) falconctl -g --apd (proxy disabled?) 3. If using proxy: falconctl -s --aph=proxy.company.com --app=8080 4. Test connectivity: curl -v https://ts01-b.cloudsink.net:443 5. Check for SSL inspection/MITM proxy (must exempt CS traffic)

HIGH False positive blocking a business application

CAUSE: ML or behavioral rule flagging legitimate app FIX (in order of preference): 1. Console > Activity > Detections > Find the detection 2. Click "..." > "Allow" to whitelist that specific file hash 3. OR: Configuration > Prevention Policy > ML Exclusions → Add file path exclusion: C:\CompanyApp\* 4. OR: Lower ML sensitivity for that host group only → Create a separate prevention policy with lower ML → Assign to the affected host group 5. For IOA false positives: → Modify the IOA rule to exclude the specific process → Add condition: ParentImage NOT contains "companyapp" IMPORTANT: Document all exclusions. Keep them minimal. Too many exclusions = security gaps.

MEDIUM Sensor installation fails

CAUSE: Prerequisites not met or conflicting software FIX (Windows): 1. Check prerequisites: .NET 3.5+, TLS 1.2 enabled 2. Check for conflicting AV: Temporarily disable other AV 3. Check install log: %TEMP%\CrowdStrike\*.log 4. Ensure admin rights for installation 5. If MSI fails: Try EXE installer or vice versa 6. Check Windows Event Log > Application for CS errors FIX (macOS): 1. Grant Full Disk Access + System Extension via MDM 2. Or manually: System Preferences > Security > Allow CrowdStrike 3. Network Filter approval required (macOS 11+) FIX (Linux): 1. Check kernel version against support matrix 2. Ensure running as root/sudo 3. Check: /var/log/falcon_install.log 4. For RHEL: Ensure proper SELinux context

INFO How to uninstall the sensor

NOTE: You need the Maintenance Token from the console Get Token: Console > Host Setup > Sensor Update Policy > Reveal Maintenance Token Windows: CsUninstallTool.exe /quiet MAINTENANCE_TOKEN=<token> OR: msiexec /x WindowsSensor.exe /quiet MAINTENANCE_TOKEN=<token> macOS: sudo /Applications/Falcon.app/Contents/Resources/falconctl uninstall --maintenance-token <token> Linux: sudo /opt/CrowdStrike/falconctl -d -f --maintenance-token <token> sudo dpkg --purge falcon-sensor (Debian/Ubuntu) sudo yum remove falcon-sensor (RHEL/CentOS)
